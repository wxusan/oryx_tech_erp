## Current local implementation evidence

- Scope implemented in this branch: E2-009 through E2-015. E2-001 through E2-008 remain `OPEN` and are not represented as delivered.
- Unit/source suite: 175 files, 1,523 tests passed, 0 failed, 0 skipped/todo.
- PostgreSQL integration suite: 11 files, 74 tests passed, 0 failed.
- Database rehearsal: all 37 Prisma migrations applied to an empty PostgreSQL database; `prisma validate` and client generation passed.
- Application gates: TypeScript typecheck, ESLint, Next.js 16.2.9 production build, and `git diff --check` passed. The build generated 56 static pages.
- Dependency audit: `npm audit --omit=dev` reports six moderate advisories through Prisma's development server dependency and Next.js's bundled PostCSS. The suggested `--force` remediation would perform breaking framework downgrades, so it was not applied. The Hono `serveStatic` path is not exposed by the application runtime, and the PostCSS path is build-time in this application, but both remain tracked supply-chain findings rather than being marked resolved.
- Seed reliability: the disposable demo database seeded successfully and then completed two consecutive guarded reset-and-reseed runs.
- Browser verification: super-admin shop creation, owner login, restricted worker creation/login, permission-filtered navigation and data, denied direct routes, and a 390 Ă— 844 mobile layout were exercised locally. Captured console errors were empty and measured page widths did not overflow the viewport.
- Performance architecture: the existing two-minute React Query cache, ChangeEvent delta sync, cross-tab broadcast, and targeted invalidation remain in place. ERP 2.0 adds member/package/permission versions to cache scope and refreshes only authorization context on access changes; Redis was not added.
- Pending external evidence: a real Telegram send matrix, preview-deployment network/Web Vitals comparison, Vercel deployment, production migration, and production browser/database smoke tests.
- Release state: these changes are in the local `codex/erp-2-0` working tree. They are not on `main` and are not online in production until the protected release and data-migration steps are separately approved and completed.
